### 1. Karpathy 提出 Software 3.0 范式：从 Vibe Coding 到 Agentic Engineering

- **来源**：AI Builders Digest（Training Data 播客）
- **价值等级**：高
- **情报判断**：Andrej Karpathy 在 Training Data 播客中抛出了一个反直觉的坦白：他从未像现在这样感觉自己是个落后于时代的程序员，而这是他最兴奋的时刻。原因不是技术退步——他发现自己已经不记得上一次需要手动修正 AI 生成代码是什么时候了。他由此提出三层软件范式：Software 1.0（手写代码）、Software 2.0（数据集+神经网络）、Software 3.0（LLM 作为可编程计算机，编程手段变成 prompt，context window 变成操控解释器的杠杆）。他举了一个关键例证：安装 OpenClaw 时，传统方式需要写复杂的平台适配 shell 脚本，但在 Software 3.0 中只需复制一段 text 给 AI agent，它就能自动完成安装。这不是渐进改进，而是计算范式的根本转变。Karpathy 的结论是：真正相信 LLM 就是新计算机的团队，会用完全不同的方式构建产品——不是写代码，而是设计 agent 行为和 prompt 系统。

  这对读者的 Agent 工具链建设有直接指导价值。当前 Agent 工具链的核心工作——skills 编写、rules 配置、上下文工程、sandbox 配置——本质上就是在做 Software 3.0 范式的"系统编程"。读者需要审视当前的工作流：有多少环节还是 Software 1.0 思维（手写脚本来解决问题），有多少已经切换到 Software 3.0（定义 prompt 和 agent 行为来替代脚本）。具体建议：对 Agent 工具链中每一条 skills 和 rules 进行"范式审计"——如果一条规则的本质是在教 agent 如何调用一个 shell 脚本来完成工作，它可能应该被重新设计为直接教授 agent 如何用自然语言完成同一件事，让 LLM 自己决定执行路径。

## 对自研引擎 / Agent 的影响

### 自研引擎

- **评估本地推理模型的可行性**：MiMo 的 1000tps 和 Apple 转向 Gemini 两条信息一致指向——外部 API 虽方便但长期依赖风险上升（定价、模型切换、稳定性）。自研引擎应建立模型推理的本地化评估 pipeline：在当前开发机上测试主流开源模型（Qwen、DeepSeek、CodeLlama）的 GGUF 量化的代码生成质量与延迟，确定本地推理是否能覆盖 80% 以上日常场景。不必追求全面替换，而是建立"本地为主 + 云端兜底"的双轨制。

- **建立 AI 能力抽象层**：借鉴 Apple 的 Core AI Framework 思路，在自研引擎工具链中构建一个模型无关的 AI 调用层。当前引擎的自动化测试、渲染验证、日志分析等环节都依赖模型输出，但每个环节直接写死了模型选择。应在这些环节的 prompt 调用前加一层 model routing 中间件，使得切换模型只需改配置，不改代码。这条建议优先级最高，因为它可以即刻减少后期模型切换带来的重构成本。

### AI Agent / Coding Agent

- **对 Agent 工具链进行"Software 3.0 范式审计"**：对照 Karpathy 的三层范式，逐条审查当前 Agent 工具链中的 skills 和 rules。规则大致分三类：(A) 本质上是 shell 脚本指导的（Software 1.0），(B) 是让 agent 调用特定 API/框架的（Software 2.0），(C) 是纯粹用 prompt 描述行为和目标的（Software 3.0）。目标是逐步将 (A) 和 (B) 类规则迁移到 (C) 类，让 agent 自己决定执行路径而不是被手把手教每一步。这项审计不需要一次性完成，而是在每次使用相应 skills 遇到问题时顺带重构。

- **建立 agent 平台中立的 skills 编写规范**：FrontierCode 的出现和 OpenAI 的 IPO 都提示一个问题——今天用得最好的 agent 平台可能不是半年后最好的。skills 的内容应面向"需要完成的任务"而不是"当前 agent 的特有指令"。例如：不写"用 Claude Code 的 /review 命令做代码审查"，而写"代码审查的标准流程：检查逻辑正确性、边界条件、性能开销、与现有架构的一致性"。后者在任何 agent 平台上都能被理解执行。

- **将实施复杂度的认知转化为工具链设计原则**：Levie 判断 agent 实施的工作量将远超想象——这对 Agent 工具链的设计者是利好而非利空。意味着当前投入在 skills 编写、规则配置、记忆系统维护上的时间不是技术负债，而是构建未来知识产权的积累。建议为此建立系统化的技能管理流程：记录每次使用 agent 时的有价值发现，定期收敛到 skills 中，形成"使用-发现-固化"的正循环。
