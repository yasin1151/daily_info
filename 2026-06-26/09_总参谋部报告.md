### 4. Codex 史诗级 BUG：高频日志写入可损耗 SSD

- **来源**：V2EX
- **价值等级**：中高
- **情报判断**：Codex 在流式任务和长时间运行时，以极高频率向 ~/.codex/logs_2.sqlite 写入 TRACE 日志，强度足以直接损耗消费级 SSD。临时方案是用 SQLite trigger 拦截 logs 表的 INSERT。X 站和 GitHub Issue (#28224) 仍在讨论修复中，多位用户确认 Mac 版和 Windows 版均中招，最新版仍未完全修复（有用户已累积 27 万次写入）。这不是功能性问题，而是硬件损伤级别的 BUG。对 Agent 工具链的直接启示：任何长时间运行的 AI coding CLI 工具都应检查日志策略；读者可对高频使用的 Agent 工具加入磁盘写入监控——先用 `iotop` 或 `lsof` 检查日志文件的写入频率，必要时配置日志轮转或定向到内存文件系统。这也提醒自研引擎开发者：在高频 IO 场景下，SQLite/文件日志的热路径设计必须考虑 SSD 寿命，尤其是在频繁的 Agent 循环写入场景中。

## 对自研引擎 / Agent 的影响

### 自研引擎

- **非编程环境训练提升编程能力的跨域泛化效应，可直接迁移到工具链验证**：让 Agent 在复杂非编码工作流（多源信息交叉验证、文档脱敏、MCP 编排、数据管线管理）中运行，这些暴露会反向提升 Agent 在编码场景的能力。在有条件时，建立包含非编码任务的"Agent 生存训练"场景集。
- **高频 IO 写入的 SSD 寿命管理**：Codex BUG 提醒自研引擎在任何高频日志/中间数据写入路径上，应考虑内存映射或临时目录落到内存文件系统的方式，避免对用户硬件造成不可逆损伤。
- **MCP Tool 暴露粒度的工程设计**：200 个 Tool 一次性暴露导致模型遗漏的问题，指向一个明确的设计原则——Tool 应当分层暴露，默认只暴露高频/核心 Tool，低频 Tool 按需加载。这适用于自研引擎的任何 API/Tool 接口设计。

### AI Agent / Coding Agent

- **缓存命中率是 Agent 成本的第一优化目标**：300 亿 Token 报告中 95.77% 缓存命中率证明，长期 Agent 工作流中缓存策略比模型选择对成本影响更大。建议对每个 Agent 会话检查缓存状态，优化 rules/AGENTS.md/skill 引用路径，确保高比例热缓存。
- **多 Agent 并行审查架构进入可行区间**：2 CC + 2 Codex 并行审查和 Cotal/A2A 协调层的出现，意味着多 Agent 不再只是实验。读者可尝试在代码 Review 阶段引入一个"审查者"角色，使用更便宜的模型做第一遍扫描，仅在发现问题时升级到强模型复查。
- **AGENTS.md 作为"路由"的设计范式**：V2EX 社区讨论形成了 Agent 工作流建设的共识——AGENTS.md 负责路由（告诉 AI 去哪里读什么文件），规范文件放在独立文件夹，skill 负责可复用能力。读者应检查自己的 Agent 工作流是否已经建立这种分层架构。
- **模型选择应基于任务类型而非品牌偏好**：GLM 5.2 在 Cursor 上可免费使用、Ornith-1.0 系列声称在参数规模上的 SOTA、Sonnet 5 在性价比上的优势——模型战场正在快速分化，Agent 工作流应预留"模型路由"层，对不同任务（规划、编码、审查、调试）分配不同模型。

### 交叉影响

- **Agent 权限隔离应加入当前工作流**：Levie 的 Agent 权限模型与 Swyx 的"重建基础设施"论指向同一方向——Agent 不是"用户账号的延伸"而是"新一类系统用户"。无论是使用 API key 隔离、角色权限文件控制、还是通过 profile 隔离，都应在当前以数据安全为前提的 Agent 工作流中落地。
- **NVIDIA Nemotron-TwoTower 的并行解码方向值得关注**：虽然社区对其反应偏冷（高赞评论"不感兴趣"），但其 2.42 倍吞吐提升的架构方向与 Agent 场景对低延迟的追求一致。如果扩散式语言模型走向成熟，Agent 的实时响应时间有望进一步压缩。应跟踪开源进展。
