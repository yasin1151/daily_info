### 3. Codex 深度使用：项目经验到底应该沉淀在哪里？

**帖子摘要**：作者系统性地使用 Codex 辅助开发，方向不是"让 AI 自动写代码"，而是希望 AI 逐渐积累项目上下文——不再每次都像新来的开发者。核心困惑：项目经验沉淀在哪里（README, AGENTS.md, docs, rules, 独立知识库）？哪些适合 AI 自己总结、哪些必须人工？skill 适合沉淀什么？MCP 怎么接真实项目？评论区给出了比较完整的答案。

**为什么值得关注**：这个问题不是理论讨论，而是大量实际使用后才遇到的。评论区总结的关键路径是：从上层 workspace 进入而不是从单个项目进入，让 Agent 先通过 subagent 理解多个项目之间的关系和责任边界，再沉淀成组织级 skill。AGENTS.md 负责"路由"（告诉 AI 去哪里读什么文件），规范文件放在独立文件夹，skill 负责可复用的能力和脚本。有用户提到 trellis 作为渐近式知识库框架也很好用。这条帖子的讨论质量很高，代表了 coding agent 工作流建设的最新共识。

**精选评论**：
- 回复 2（最高信息密度）：把规范文件放在独立文件夹，AGENTS.md 当路由（每个文件对应什么职责），每次写代码前先让 AI 出实施计划，review 完发现缺少规范就补充规范文件夹，再重新定计划，直到计划完美——几轮下来，规范文件夹和 AGENTS.md 就是基于项目深度定制的。
- 回复 4（上层 workspace 进入）：建议从上层目录进入，让 Agent 通过 subagent 搞懂每个项目后再定位项目之间的关系，最后沉淀成产品级 skill。
- 回复 9（skill 和 AGENTS.md 的区别）：指出 AGENTS.md 各个 Agent 文件名不规范、不便迁移；skill 下有文档、脚本、tools，可用 npx skills 统一管理，通过 git 团队同步。
