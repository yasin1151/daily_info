### 1. Go 跨平台打包：把服务端塞进客户端是否合理？

**摘要**：讨论源于一个 Go 客户端把服务端能力一并打包，实现本地离线模式。核心争议不只是 Go 打包方便，而是桌面应用在原生、跨平台绘图库、WebView/Electron/Tauri 之间如何权衡包体、体验、维护成本和离线能力。

**高质量评论**：
- **13 楼｜点赞数未公开**：把 UI 技术路线拆成三类：系统原生绘制、Skia/Avalonia/Flutter 这类跨平台绘图库、以及 Electron/WebView。指出核心差异在布局与控件绘制控制，跨平台方案便利但通常要牺牲包体和生态成熟度。
- **5 楼｜点赞数未公开**：从 .NET/Avalonia 角度补充，强调 solution/工程管理、统一构建、WinUI 原生 API 兼容、macOS 开发、Kestrel 性能和 async/await 支持，认为 C# 做 GUI 仍有强理由。
- **6 楼｜点赞数未公开**：提出工程落地问题：Avalonia 打包出的 Windows exe 体积多大？自己曾因 .NET 包体过大改用原生 Win32 API，实现了一个轻量工具。

### 2. 用 AI 写代码时，还要逐行看代码吗？

**摘要**：楼主认为 AI 生成大量后端、接口、SQL 和测试后，逐行 review 的收益下降，自己转向“结果验收”：明确需求、补测试、跑页面、验业务口径和失败日志。这反映 AI 编程从看代码转向验系统行为的工作流变化。

**高质量评论**：
- **32 楼｜点赞数未公开**：认为 AI 一天产出的代码量人类根本看不过来，建议让 AI 维护 claude.md、测试、注释、文档；每次提交先 AI review，人只检查 review 结果和 UI/业务逻辑。
- **38 楼｜点赞数未公开**：持谨慎态度，建议控制 AI 一次不要生成上百行，而是小功能逐步写；否则多线程等场景里的微妙逻辑 bug 会很难 debug。
- **42 楼｜点赞数未公开**：提出折中视角：不逐行看代码，但要看数据结构、接口定义、代码结构和架构设计。

### 3. Claude 被封后改用 GLM：国产模型真的更便宜吗？

**摘要**：楼主在 Claude 被封后用 GLM，发现一天花费 21 元，质疑国产模型并不便宜。评论焦点在 API 计费与 coding plan 不能直接比较，以及 GLM、Claude、OpenRouter、opencode 等代理/订阅模式的真实成本差异。

**高质量评论**：
- **5 楼｜点赞数未公开**：指出第三方订阅套餐可提供 GLM 5.2 使用权限，例如 opencode go 首月 5 美元、续订 10 美元，并含约 60 美元开源模型额度，不能只按官方 API 单价判断。
- **8 楼｜点赞数未公开**：解释楼主真实痛点：习惯 Claude Code 的 agent harness 和交互方式，即便外接 GLM，也会受工具链习惯、套餐缺货、协议端点等因素影响。
- **33 楼｜点赞数未公开**：给出成本对比样例：GPT-5.5 medium 约 20k 输出 token、46 steps、均价 2.75 美元；GLM-5.2 max 约 54k 输出 token、122 steps、均价 3.92 美元。提示“步骤数”会放大实际成本。

### 4. Claude 封号潮：模型体验、账号风险与工作流迁移

**摘要**：楼主被封 Claude 后回看历史聊天，认为 Claude 在日常对话和文字表达上难被 Gemini、ChatGPT、Codex 替代。帖子背后是一个更大的问题：AI 工具已成为情感、写作和编程工作流的一部分，但账号风控让依赖关系变得脆弱。

**高质量评论**：
- **23 楼｜点赞数未公开**：用类比概括 Claude 与 Codex 的差异：Claude 体验好但风控不稳定；Codex 更像稳定可用的工具，额度和可持续性更让人安心。
- **83 楼｜点赞数未公开**：来自多模型用户视角：同时用 GPT Pro、Claude API、GLM coding plan 后，认为当前 Claude 并非所有场景都遥遥领先；是否依赖它，很大程度取决于是否有“情感/表达”需求。
- **25 楼｜点赞数未公开**：指出通过 Cursor 使用 Claude 会受到默认系统提示词影响，输出更偏编程，不一定有官网 Claude 的“味道”；而官方 API 又成本过高。
