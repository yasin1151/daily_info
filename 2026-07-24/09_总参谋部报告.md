### 2. Echo——用开源模型池达到 Fable 级别效果，成本仅 1/3

- **来源**：HackerNews Show HN
- **一句话人话版**：有人在 HN 发布了一个多模型路由系统 Echo，用一批开源模型混合编排达到顶级闭源模型的效果，成本只有三分之一。
- **新闻/帖子本身**：Echo 是一个多模型路由/编排系统，将一批开源权重模型（GLM-5.2、Kimi K2.7 等）组成模型池，对每个请求动态决定需要多少算力、哪些模型参与、如何组合输出。核心洞察是不同模型在不同子任务上各有专长，整体表现可以超过池中任何单个模型。作者宣称 Echo 持续优于池中最佳模型，达到 Fable 的聚合水平但推理成本仅 1/3。提供 OpenAI 兼容 API 和聊天界面。
- **社区原话摘录**：
  - HN 评论者（正方）：认为是 ensemble / mixture-of-experts 在推理层的合理应用——"if you can route to the right model for each subtask, you get better results than a single monolithic model"（如果能针对每个子任务路由到合适的模型，结果会比单一大型模型更好）
  - HN 评论者（反方）：质疑基准测试不够硬核——"show me the numbers on something like SWE-bench or HumanEval, not cherry-picked examples"（在 SWE-bench 或 HumanEval 上展示数据，而不是精心挑选的示例）
  - HN 评论者（中立）：认为这更像是 OpenRouter Fusion 或 Sakana Fugu 的变体，不是真正的新方法——"routing has been done before, the novelty is in the claim of matching Fable at 1/3 cost"
- **参谋部短评**：低成本路由+开源模型对抗高价 API 的商业模式值得关注，对做 gateway/middleware 的团队有直接参考价值。但核心问题在于基准测试的透明度——Echo 的"达到 Fable 水平"是真是假，需要独立验证。

### 3. Claude 开始自创术语和缩写，用户被迫当"猜谜者"

- **新闻/帖子本身**：多位用户反映 Claude（尤其在 Fable 和 Opus 模型上）开始频繁自行创造术语、缩写和内部编号——例如把决策点标记为"D32"、缩略清单条目为"V-3-2"等，然后用这些自创编号来回引用。用户不得不频繁追问"你说的 X 是什么意思"。问题在长会话和复杂项目语境中尤为突出：Claude 似乎为了节省 token 而采用"内部编码"式缩写，但对人类用户完全不透明。有用户尝试用 Fable 自动生成一个将自创 code 链接到源文件的 linker 脚本。
- **社区原话摘录**：
  - Reddit u/jeffcabbages（37 赞）："I have also noticed an uptick in the necessity to ask it 'what do you mean by...' and it needed to explain the weird title it came up with on its own." —— 这已成为一个加剧的趋势：用户需要花越来越多的时间去追问 Claude 自己的术语含义

  - Reddit u/obsidience（3 赞）："I had Fable build a linker script so that any references to these codes in the MD documentation get automatically linked to the source file that created and explains the code reference." —— 一位用户选择利用而非对抗：让 Fable 自动维护跨文档引用解析脚本，将自创 code 链接到源文件
- **参谋部短评**：这是模型行为退化值得关注的信号——节省 token 的设计决策正在损害可用性。对于重度 Claude Code 用户，建议在系统提示中加入"不要自创术语，除非你已经在词汇表中定义并维护"。
